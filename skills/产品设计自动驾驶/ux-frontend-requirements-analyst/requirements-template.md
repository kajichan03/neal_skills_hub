# 需求分析文档模板

输出路径：`.workflow/requirements/<feature>.md`

---

## 模板

```markdown
# <Feature> 需求分析

生成时间：<YYYY-MM-DD>
输入源：<标注实际使用的来源，三选一>

- 设计稿路径：`.workflow/design-from-figma-make/<dir>/`
- PRD 路径：`.workflow/prd/<file>`
- 设计稿 + PRD（路径同上）

---

## 1. 产品背景

<用 2-4 句话描述：这个功能解决什么问题、目标用户是谁、核心价值是什么。

- 有 PRD 时：基于 PRD 的功能目标重新表述
- 仅有设计稿时：摘自 Project_Goal.md，用自己的理解重新表述
- 两者都有时：以 PRD 为主，设计稿补充>

**功能性质判断**：

- 全新功能：是 / 否
- 判断依据：<一句话说明来源，如“PRD 第 X 节明确标注为新模块”或“业务域 '×××' 在现有 codebase-map 中无匹配”>
- 建议插件类型：新建前端业务插件 `plugins/<domain>` / 扩展现有插件 `plugins/<existing>` / 新建后端插件 `plugins/<domain>-backend` / 无需新建
- 建议插件名（如需新建）：`<domain>`（kebab-case，与 feature slug 保持一致）

---

## 2. 功能模块路径

<例：变更管理 / 泳道管理 / 泳道详情页>

---

## 3. 页面关系（多页面时必填）

<描述各页面之间的关系：

- 入口页面是哪个，从哪里可以导航到这里
- 页面间如何跳转，携带哪些参数
- 哪些数据/组件在多个页面间共享
  如果只有一个页面，写"单页面，无跨页面跳转"。>

---

## 4. 页面整体结构

<用树形结构描述页面的区域划分，要基于设计稿 App.tsx 的实际 div 嵌套层次，而不是子组件的命名。例：

页面（单白卡）
├── 头部区域
│ ├── 左侧：标题 + 状态标签 + 副标题信息
│ └── 右侧：操作按钮组
├── 概览区域（二列信息网格）
├── 表格区域 A
├── 表格区域 B
└── 图表区域（两列并排）

>

---

## 5. 各区域详细说明

> 每个区域一个小节，按页面从上到下的顺序排列。

### 5.x <区域名>

**作用：** <这个区域的业务职责是什么>

**展示内容：**
<列出该区域展示的所有字段：

- 字段名：来源（API 字段路径）/ 展示规则
- 例：泳道名称：来自 lane.name，无数据时显示"-"
- 例：状态标签：来自 status 字段，不同值对应不同颜色——详见第 6 节状态映射>

**操作：**
<列出该区域包含的可操作元素：

- 按钮/链接名称：触发什么动作（打开弹窗/跳转/API 调用）
- 例：手动销毁按钮：点击后打开确认弹窗，确认后调用删除接口，成功/失败均有消息提示>

**空状态：** <当数据为空或加载失败时该区域如何表现>

## 6. 状态与值的映射规则

<集中描述页面中所有"值 → 展示"的映射关系，避免分散在各区域重复描述。例：

### 泳道状态（status 字段）

| 值             | 显示文案 | 标签颜色 |
| -------------- | -------- | -------- |
| ACTIVE         | 运行中   | 绿色     |
| DEPLOYING_APPS | 部署中   | 蓝色     |
| FAILED         | 失败     | 红色     |
| FINISHED       | 已完成   | 灰色     |
| 其他           | 原始值   | 默认     |

### 中间件状态（middlewareStatus 字段）

（同上格式）

>

## 7. 弹窗与对话框

> 每个弹窗一个小节。

### 7.x <弹窗名>

**触发：** <哪个按钮/操作打开此弹窗>

**内容：**
<描述弹窗里有什么：提示文案、列表、表单、按钮等>

**交互流程：**
<描述用户在弹窗里能做什么，操作后发生什么，完全用文字描述，例：

1. 用户看到所有关联组件的链接列表，可点击各链接跳转到对应页面
2. 用户点击"重启"按钮，按钮变为 loading 状态，禁用点击
3. 2 秒后（或 API 回调后），显示成功/失败提示，按钮恢复可用>

**后端依赖：** <需要哪个 API，目前是否已实现；如未实现注明"后端待实现，当前用 Mock">

## 编写原则

**目标：让 LOCATE 阶段和 PLAN 阶段在不读设计稿的情况下能完整工作。**

1. **用自然语言描述业务逻辑** — 不写代码，不写 JSX，不写 TypeScript 接口定义；用"当用户点击 X，发生 Y，结果是 Z"这样的句式
2. **字段描述要精确** — 字段的 API 路径、空值规则、格式化方式都写清楚；不要写"显示状态"，要写"显示 status 字段的值，ACTIVE 显示绿色'运行中'，其余见状态映射表"
3. **交互描述要完整** — 每个按钮的触发条件、操作流程、成功/失败的结果都要写；不要遗漏 loading 状态、禁用条件、页面跳转
4. **Mock 说明要标注** — 哪些数据是 Mock、Mock 的形式是什么、未来如何替换，集中在第 11 节
5. **不含实现决策** — 不写"用 Ant Design Table"、不写"用 useState 管理 loading"；这些属于 PLAN 阶段决策
6. **严格按 Section 顺序** — 1-11 全部填写，确实不适用时写"不涉及"而非留空
```
