# When to use

在以下情况使用此 skill：
- 已完成 问题锚定，拥有 `00-quick-anchor.md`
- 需要将模糊的问题转化为具体的用户故事
- 需要调研业界现有解决方案作为参考

# When NOT to use

在以下情况不应使用此 skill：
- 问题尚未锚定，仍在探索阶段 → 应使用 `/ux:phase0-anchor`
- 已有明确的解决方案，只需执行 → 应直接进入后续阶段
- 需要进行多方案对比评估 → 应使用 `/ux:phase3-synthesis`

---

## 概述

**目标**：基于已锚定的问题和补充的上下文，编写用户故事并调研业界现有解决方案，为后续 Synthesis 提供完整输入。

**输入**：
- `00-quick-anchor.md`（问题锚定 产出的草稿 Problem Statement + 争议点）
- `00-context.md`（上下文补充 补充的上下文，如已执行）

**输出**：`02-solution-research.md`（用户故事 + 业界解决方案调研）

---

## 准备工作

首先读取前置阶段的输出：

1. **必须读取**：`00-quick-anchor.md` - 获取草稿 Problem Statement 和争议点
2. **必须读取**：`00-context.md` - 获取补充的组织背景、技术栈、约束条件

如果 `00-quick-anchor.md` 不存在，提示用户：

> "建议先运行 `/ux:phase0-anchor` 进行问题锚定。是否继续？"

---

## 执行策略：两条工作流串行执行

解决方案研究 包含两条工作流，**按顺序执行**：

1. **Workstream 1：编写用户故事** - 基于问题和背景，描述如何解决问题
2. **Workstream 2：调研业界解决方案** - 调研业界现有的解决方式

> 串行原因：Workstream 2 的解决方案调研需要参考 Workstream 1 的用户故事，确保调研的方案能够覆盖用户故事描述的需求。

## Workstream 1：编写用户故事

> 🤖 **AI 角色：用户故事编写者**

**核心任务**：根据已有的 Problem Statement、争议点和背景信息，编写完整的用户故事，描述如何解决这个问题（功能或流程）。

**支持多方案**：一个问题可以有多个可选的解决方案，对应多个用户故事。

### 用户故事三要素

每个用户故事必须包含：

| 要素 | 说明 | 示例 |
|-----|------|------|
| **角色（Who）** | 谁要使用这个功能 | 作为一个系统管理员 |
| **活动（What）** | 要完成什么活动 | 我想要一键同步所有数据中心的 Redis 配置 |
| **价值（Value）** | 为什么要这么做，带来什么价值 | 以便于减少重复配置工作，降低配置错误风险 |

### 编写顺序

**按 1→2→3 的顺序思考和编写**：

1. **先定义角色（Who）** - 基于 问题锚定 识别的用户类型
2. **再明确价值（Value）** - 解决什么痛点，带来什么收益
3. **最后设计活动（What）** - 具体的功能或流程

### 引导用户确认

使用 **AskUserQuestion tool** 与用户协作编写：

> "基于我们锚定的问题：'[引用 PS]'
>
> 让我们来编写用户故事：
> 1. **角色**：谁是这个功能的主要使用者？（参考 问题锚定 识别的用户类型）
> 2. **价值**：解决这个问题能给他们带来什么具体收益？
> 3. **活动**：他们需要完成什么具体动作来实现这个价值？"

**如果有多个解决方案方向**，逐一讨论：

> "这个问题可能有多个解决方向。除了刚才讨论的方案，还有其他可能的解决方式吗？"

## Workstream 2：调研业界解决方案

> 🤖 **AI 角色：解决方案调研员**

**核心任务**：通过已有知识或搜索网页，调研业界现有哪些解决方案。

**支持多方案**：一个问题可能有多个业界解决方案，逐一调研记录。

### 调研内容格式

每个业界解决方案也采用类似用户故事的结构，但在 **活动（What）** 部分需要给出具体描述：

| 要素 | 说明 |
|-----|------|
| **角色（Who）** | 该方案服务的目标用户类型 |
| **活动（What）** | 方案提供的具体功能、方法、流程（A → B → C） |
| **价值（Value）** | 该方案宣称能带来的价值 |

### 活动（What）部分的具体要求

必须包含以下信息：
- **提供了什么功能/方法**：具体的能力描述
- **具体流程**：用户需要如何操作（步骤 A → 步骤 B → 步骤 C）
- **参考链接**：官方文档、GitHub 仓库、产品介绍页面等

### 调研来源

使用以下方式获取信息：
1. **已有知识** - 模型训练数据中的相关方案
2. **搜索网页** - 使用 WebSearch 工具搜索最新方案

### 三类替代方案扫描

确保调研覆盖：

1. **开源方案**
   - GitHub 上的相关项目
   - 开源工具/框架
   
2. **商业产品/SaaS**
   - 可直接采购的解决方案
   - 价格/集成门槛信息
   
3. **用户的"笨办法"**
   - Excel 登记、Wiki 文档、口头传递等
   - 这些方法的崩溃点和隐性优势

## 护栏

- **先用户故事，后方案调研** - 两条工作流必须串行，确保调研有针对性
- **用户故事三要素缺一不可** - Who + What + Value 必须完整
- **方案调研要具体** - 活动部分必须有流程描述和参考链接
- **笨办法是竞品** - 必须包含用户现有土办法的分析
- **支持多方案并行** - 一个问题可以有多个用户故事和多个业界方案
- **打待验证标记** - 不确定的信息标注清楚，留给 对抗式 Synthesis 讨论
