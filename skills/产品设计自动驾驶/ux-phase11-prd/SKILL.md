# Phase 11：产品研发需求文档（PRD）生成

## 概述

**目标**：整合项目章程中的背景、需求、规则和原型方案，生成一份**面向研发和测试**的、**结果导向**的 PRD 文档。

**输出要求**：生成的 PRD 文档必须在开头提供目录（Table of Contents），便于快速导航。

## 核心原则

| 原则 | 说明 |
|------|------|
| ✅ **结果导向** | 只保留最终结论，去除调研/讨论过程 |
| ✅ **研发友好** | 提供足够的背景信息、清晰的需求描述、明确的业务规则 |
| ✅ **测试友好** | 规则清晰，便于编写测试用例 |
| ✅ **目录导航** | 文档开头必须提供完整的目录，便于快速定位 |
| ❌ **无过程信息** | 不包含调研记录、讨论过程、决策理由 |

## 你的角色：PRD 文档工程师

你的任务是将 Project Goal 中的**结果性信息**提取并重组为标准的 PRD 格式。

### 与 Project Goal 的区别

| 维度 | Project Goal | PRD |
|------|--------------|-----|
| 内容 | 包含过程信息（🔍 标记、复杂度评估、逐步推导） | 只包含最终结果 |
| 读者 | 产品团队内部 | 研发和测试团队 |
| 目的 | 记录决策过程 | 指导开发实现 |

## PRD 核心原则

### 1. 结果导向 — 去除过程信息

| 信息类型 | Project Goal 可能包含 | PRD 保留方式 |
|---------|---------------------|-------------|
| 需求来源 | 调研过程、用户访谈记录 | ❌ 移除 |
| 规则推导 | 逐步分析过程、🔍 标记 | ❌ 移除 |
| 决策过程 | 为什么选择这个方案 | ❌ 移除 |
| 讨论记录 | Phase 3 对抗式辩论 | ❌ 移除 |
| **最终背景** | — | ✅ 完整保留 |
| **最终需求** | — | 完整保留 |
| **最终规则** | — | 完整保留 |
| **原型链接** | — | 完整保留 |

### 2. 研发友好 — 信息完整清晰

每个功能点必须包含：

- **背景** — 这个功能解决什么问题
- **需求描述** — 具体要做什么
- **业务规则** — 明确的逻辑和条件
- **数据说明** — 涉及的数据字段和类型
- **验收标准** — 可测试的通过条件

### 3. 测试友好 — 规则可验证

- 规则用表格呈现，条件明确
- 状态机有完整的流转定义
- 每个功能点有清晰的验收标准

### 4. 目录导航 — 快速定位

PRD 文档开头**必须**包含目录（Table of Contents），包含：
- 一级标题（章节）
- 二级标题（子章节）
- 可选：三级标题（功能点清单）
- 链接到对应章节

## 提取规则

### 从 Project Goal 提取时，去除以下内容

| 去除内容 | 示例 | 原因 |
|---------|------|------|
| 调研过程 | "通过 5 轮用户访谈发现..." | 过程信息 |
| 讨论记录 | "Phase 3 中 Tim Cook 质疑..." | 过程信息 |
| 复杂度标记 | "此功能点标记为 🔍" | 过程标记 |
| 决策理由 | "选择方案 A 而非方案 B 是因为..." | 决策过程 |
| 逐步推导 | "首先分析...然后得出..." | 推导过程 |

### 只保留最终结果

| 保留内容 | 来源 |
|---------|------|
| 项目背景（最终版） | Project Goal 第 2 章 |
| 目标用户（最终版） | Project Goal 第 1 章 |
| 需求描述（最终版） | Project Goal 第 3 章 |
| 功能点规则（完整版） | Project Goal 第 3 章（含 Phase 8 补充）|
| 原型方案链接 | Project Goal 第 3 章 或 Phase 10 补充 |

## 执行步骤

### Step 1：读取 Project Goal

读取 `07-project-goal.md`，识别：
- 项目背景和目标
- 角色定义
- 需求清单和功能点
- 详细规则（Phase 7 简单规则 + Phase 8 补充规则）
- 原型方案链接

### Step 2：提取结果信息

**保留**：
- [x] 背景概述（第 2.1 节）
- [x] 目标用户（第 1 章）
- [x] 项目目标（第 2.2 节）
- [x] 成功指标（第 7 章）
- [x] 需求名称和用户故事（第 3 章）
- [x] 功能点规则（第 3 章功能点下的规则表格）
- [x] 原型链接（第 3 章功能点下的原型方案）

**去除**：
- [ ] 所有 🔍 标记和复杂度说明
- [ ] "Phase X 识别"、"调研发现"等过程性描述
- [ ] 决策理由和方案对比

### Step 3：重组为 PRD 格式

按照 PRD 结构重新组织内容：
1. 项目背景（简洁）
2. 功能需求（详细，按需求-功能点层级）
3. 页面结构（清晰）
4. 非功能需求（补充）
5. 附录（术语等）

### Step 4：生成目录

**重要**：在 PRD 文档开头生成目录，包含：
- 所有一级标题（#）
- 所有二级标题（##）
- 链接到对应锚点
- 格式示例：
  ```markdown
  ## 目录
  
  - [1. 项目背景](#1-项目背景)
    - [1.1 背景概述](#11-背景概述)
    ...
  ```

### Step 5：补充验收标准

为每个功能点添加验收标准：
- 基于业务规则编写可测试的验收项
- 每个规则对应至少一个验收标准

使用 **AskUserQuestion tool** 确认：

> "已为 FUNC-001 编写以下验收标准：
> - [ ] 字段A为空时，提交按钮禁用
> - [ ] 状态为"已完成"时，显示绿色标记
>
> 是否准确？是否需要补充？"

## 护栏

| 规则 | 说明 |
|------|------|
| **必须去除过程信息** | PRD 不是过程记录，是结果交付物 |
| **必须保留完整规则** | 研发和测试需要清晰的业务逻辑 |
| **必须有验收标准** | 每个功能点都要有可测试的通过条件 |
| **必须可追溯** | 每个需求/功能点都能追溯到 Project Goal |
| **必须包含目录** | PRD 文档开头必须有完整的目录 |
| **保持简洁** | 不重复 Project Goal 已明确的信息 |
