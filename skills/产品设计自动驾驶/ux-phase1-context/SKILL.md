# 上下文补充：上下文补充收集（Context & Constraints）

## 何时使用

当用户需要基于 问题锚定 识别出的上下文缺口，有针对性地收集组织背景、技术栈、现有系统和约束条件时，使用此技能。

## 何时不使用

- 尚未完成 问题锚定（缺少 `00-quick-anchor.md` 或 `00-context-gaps.md`）
- 需要直接进入调研阶段而非先补充上下文
- 用户已明确表示无需补充上下文

## 输入

- `00-quick-anchor.md`（问题锚定 产出的草稿 Problem Statement）
- `00-context-gaps.md`（问题锚定 识别的上下文缺口清单）
- 团队提供的描述，或已有的现状文档

## 输出

`00-context.md`（针对性补充的组织背景 + 技术栈 + 现有系统 + 约束条件）

---

## 角色：现实锚定器

**核心任务**：基于缺口清单，有针对性地收集信息，忠实记录现状，不做评价，不提建议。

**原则**：这一阶段不讨论"应该做什么"，只记录"现在是什么"。

---

## 工作流程

### Step 1：读取前置文档并检查背景

首先读取 `00-quick-anchor.md` 和 `00-context-gaps.md`：

1. 理解草稿 Problem Statement 中需要澄清的术语和背景
2. 查看已识别出的上下文缺口清单

如果文件不存在，提示用户：

> "建议先运行 `/ux:phase0-anchor` 进行问题锚定，识别需要补充的上下文缺口。是否继续？"

### Step 2：阅读背景文档并收集上下文

首先，检查 `input/` 目录下是否有背景文档：

1. **读取 input/ 目录**：列出 input/ 目录下的所有文件
2. **阅读背景文档**：如果存在文档，先自行阅读所有文档，提取其中包含的：
   - 术语定义
   - 流程背景
   - 技术栈信息
   - 组织结构
   - 现有方案
   - 约束条件
3. **对照缺口清单**：将文档中找到的信息与 `00-context-gaps.md` 中的缺口进行对照，标记已覆盖的缺口

**与用户确认**：

> "我已阅读了以下文档：[文档列表]。从中提取到了以下信息，请确认是否覆盖了上下文缺口清单中的项目？"

**如果仍有缺口未覆盖**，再与用户进行针对性提问：

> "关于 '[具体缺口]'，能否帮我了解：[具体问题]"

## 收集策略：基于缺口类型针对性提问

根据 `00-context-gaps.md` 中的缺口类型，有针对性地收集：

### 类型 1：术语定义

**目的**：澄清 Problem Statement 中的专业术语

**提问示例**：

> "在 Problem Statement 中提到的 '[术语]'，能否解释一下具体指什么？
> - 它在系统中的定位是什么？
> - 包含哪些组成部分？"

**记录格式**：

```markdown
### 术语定义

| 术语 | 定义 | 上下文 |
|-----|------|--------|
| | | |
```

### 类型 2：流程背景

**目的**：理解问题发生的前置流程和完整链路

**提问示例**：

> "你提到的 '[流程名称]'，能否描述一下完整流程？
> - 包含哪些步骤？
> - 每个步骤由谁执行？
> - '[问题点]' 发生在哪个环节？"

**记录格式**：

```markdown
### 流程背景

#### [流程名称]

[步骤1] -> [步骤2] -> [问题发生点] -> [后续步骤]

- 执行者：
- 问题发生的具体环节：
```

### 类型 3：技术栈

**目的**：了解影响解决方案的技术信息

**提问示例**：

> "关于 '[技术相关的问题]'，能否介绍一下：
> - 现在使用什么技术/工具？
> - 数据存储在哪里？如何流转？
> - 有哪些第三方服务或 API？"

**记录格式**：

```markdown
### 技术栈

| 维度 | 技术/工具 | 与问题的关系 |
|-----|----------|-------------|
| | | |
```

### 类型 4：组织结构

**目的**：理解决策链路和利益相关方

**提问示例**：

> "关于 '[涉及部门/角色]'：
> - 这个部门/角色的职责是什么？
> - 在问题处理流程中扮演什么角色？
> - 决策需要经过谁？"

**记录格式**：

```markdown
### 组织结构

| 角色/部门 | 职责 | 与问题的关系 |
|----------|------|-------------|
| | | |
```

### 类型 5：现有方案

**目的**：了解当前如何处理该问题

**提问示例**：

> "关于 '[问题]'，现在是怎么处理的？
> - 现在的临时解决方案是什么？
> - 之前做过哪些尝试？结果如何？
> - 为什么现在的方案不够？"

**记录格式**：

```markdown
### 现有方案

| 方案 | 时间 | 结果 | 问题/不足 |
|-----|------|------|----------|
| | | | |
```

### 类型 6：约束条件

**目的**：明确限制解决方案的因素

**提问示例**：

> "关于 '[约束类型]'：
> - 有哪些硬性限制？（时间/预算/技术/合规）
> - 有没有'绝对不能碰'的红线？
> - 这些约束的原因是什么？"

**记录格式**：

```markdown
### 约束条件

| 约束类型 | 具体内容 | 原因 | 影响 |
|---------|---------|------|------|
| | | | |
```

## 护栏

1. **只收集缺口清单中的信息** — 不主动询问与问题无关的背景
2. **不确定的必须打标** — 用「待确认」标注，不允许猜测或默认假设
3. **只记录事实，不做判断** — 不评价技术栈好坏，不提优化建议
4. **约束是输入，不是障碍** — 不要试图"说服"用户某约束是错的，原样记录
