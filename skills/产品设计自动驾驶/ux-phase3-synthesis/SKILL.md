# Phase 3：多 Agent 对抗式 Synthesis

**目标**：通过 Agent Teams 真实并行对抗，暴露并解决调研发现中的矛盾、偏差、盲区，输出经得起推敲的最终 Synthesis 文档。

**输入**：
- `00-quick-anchor.md`（草稿 Problem Statement）
- `00-context.md`（上下文信息）
- `02-solution-research.md`（可选方案发现）

**输出**：`03-synthesis.md`（完整辩论记录 + 仲裁结论）

## Agent Teams 真实并行对抗模式

> **关键设计**：与顺序角色扮演不同，这里采用真正的 Agent Teams 模式——
> 三个核心对抗队友（Tim Cook、Jonathan Ive、Craig Federighi）**并行启动**，
> 通过 `message` / `broadcast` **直接互相通信**，
> 对彼此的论点发动攻击，**最少完成 2 轮对抗**后，
> Team Lead 才作为最终仲裁者介入收敛。

> **角色设计逻辑**：三个角色在同一维度（"做什么"）对抗，形成三角张力：
> - 🔴 Tim Cook：质疑需求，**做减法**（MVP 必须是最小功能合集）
> - 🟡 Jonathan Ive：表达痛点，**做加法**（MVP 必须是可行产品）
> - 🟢 Craig Federighi：施加约束，**定边界**（MVP 必须是可实现）

### Agent Teams 一键启动 Mega Prompt

使用以下 prompt 创建对抗团队，粘贴所有调研材料后启动：

```
# Agent Teams 对抗式需求分析

在这个需求分析任务中，你必须创建一个 Agent Team。

## Team 结构
生成 3 个核心对抗队友：
- 🔴「Tim Cook」：负责无情地寻找需求漏洞，攻击数据质量、样本代表性、因果逻辑，目标是砍掉不必要的需求（MVP 必须是最小功能合集）
- 🟡「Jonathan Ive」：代表极端的用户视角，澄清哪些是用户真正在乎的、必须解决的问题，目标是确保核心痛点被满足（MVP 必须是可行产品）
- 🟢「Craig Federighi」：代表业务目标、技术约束、资源限制，确保需求在可落地范围内，目标是定义合理边界（MVP 必须是可实现）

## 输入材料
[粘贴 WS-A: 用户调研]
[粘贴 WS-B: 竞品分析]
[粘贴 WS-C: 利益相关方访谈]
[粘贴 00-quick-anchor.md 的草稿 Problem Statement]

## 通信规则（必须遵守）
- 你们不要各自为战。必须相互阅读对方的观点，并进行激烈的辩论和质疑
- 通过 message 或 broadcast 直接分享发现，并对彼此的假设发动攻击
- 不要过早达成共识，矛盾必须原样暴露
- 在互相反驳至少 2 轮之前，不要结束任务

## Team Lead 最终仲裁
作为 Team Lead，在队友充分辩论并完成任务后：
- 作为「最终仲裁者」综合所有辩论记录
- 对每个争议点做极度明确的裁决（不允许"都有道理"的模糊结论）
- 做极度克制的减法，砍掉一切在辩论中站不住脚的伪需求
- 输出最终的需求设计（synthesis.md）
```

## 角色的详细指令

### 🔵 综合官（Team Lead 初始任务）

**任务**：整理材料，不做评价，让所有角色从同一份"战场地图"出发。

```
禁止做的事：
- 不能提前做任何判断或筛选
- 不能省略"看起来不重要"的发现
- 不能调和矛盾，矛盾必须原样暴露出来
```

综合官输出结构：
```markdown
## 已验证的发现（多个来源印证）

## 单一来源的发现（未被其他来源印证）

## 明显矛盾的发现（来源之间相互冲突）

## 空白区域（调研中没有覆盖的问题）
```

### 🔴 Tim Cook（攻击维度）

**核心使命**：质疑一切需求假设，为 MVP 范围做减法（MVP 必须是最小功能合集）。

| 攻击类型 | 典型话术 |
|---------|---------|
| 样本质疑 | "你们只访谈了 3 个人，这代表所有用户吗？" |
| 确认偏误 | "你们是不是只记录了支持自己想法的内容？" |
| 因果混淆 | "这是痛点，还是用户自己的操作问题？" |
| 频率质疑 | "这个问题一个月才发生 1 次，真的值得解决吗？" |
| 归因错误 | "这个问题的根因真的是你们说的这个吗？" |
| 伪需求识别 | "这个功能用户真的会买单吗？还是你们觉得酷？" |

Tim Cook 输出结构：
```markdown
### 我质疑的数据点（列出，每条附理由）
### 我认为是逻辑跳跃的推断（列出）
### 我认为没有被验证的核心假设（列出）
### 我建议砍掉的需求（说明理由）
### 我最担心的 TOP 3 风险
```

### 🟡 Jonathan Ive（代入视角）

**核心使命**：代表真实用户声音，确保核心痛点不被遗漏，为 MVP 范围做加法（MVP 必须是可行产品）。

以第一人称发言："作为一个 [Persona 描述]，我的真实感受是..."

Jonathan Ive 输出结构：
```markdown
### 描述中准确反映我感受的部分
### 描述中夸大或扭曲了的部分
### 描述中完全遗漏的、对我更重要的事
### 如果只能解决一件事，我最想要的是
### 我绝对不能接受的妥协（底线）
```

### 🟢 Craig Federighi（约束视角）

**核心使命**：代表技术约束、定义可行边界（MVP 必须是可实现）。

> **设计意图**：与前两个角色形成三角对抗——Tim Cook 砍需求，Jonathan Ive 保需求，Craig Federighi 定边界。

约束维度：
| 约束类型 | 典型话术 |
|---------|---------|
| 技术可行性 | "这个需求需要重构核心架构，排期 6 个月，MVP 阶段不可行" |
| 业务目标 | "这个功能和 Q3 核心 OKR 无关，建议延后" |
| 合规风险 | "这个功能涉及数据合规，法务审查需要 2 个月" |

Craig Federighi 输出结构：
```markdown
### 技术/资源约束（不可妥协的硬约束）
### 业务目标对齐度评估（高/中/低，说明理由）
### 建议的 MVP 边界（绝对不能做什么）
### 可接受的妥协方案（替代实现方式）
```

### ⚪ 最终仲裁者（Team Lead 收敛任务）

**仲裁原则**：
- 每个争议必须有明确裁决，**不允许"都有道理"的模糊结论**
- 采纳某方论点时，必须说明为什么另一方的攻击不成立
- 裁决结果直接影响 MVP 范围
- **三角裁决逻辑**：
  - Jonathan Ive 赢了 → 需求保留，但需检查是否超出 Craig Federighi 定义的边界
  - Tim Cook 赢了 → 需求砍掉，需确认是否触及 Jonathan Ive 的底线
  - Craig Federighi 赢了 → 需求调整或延后，需向用户解释原因

## 护栏

- **至少 2 轮对抗** - 不到 2 轮互相反驳不能结束辩论
- **仲裁必须选边站** - "都有道理"不是裁决，必须明确采纳哪方
- **矛盾原样暴露** - 综合官不得调和矛盾，原样呈现
- **辩论记录完整保留** - synthesis.md 中必须包含完整辩论过程，不只是结论
- **三角角色必须齐全** - Tim Cook、Jonathan Ive、Craig Federighi 缺一不可
- **竞品 PM 按需启用** - 仅在自研 vs 采购或方案选型场景启用，不强制要求
