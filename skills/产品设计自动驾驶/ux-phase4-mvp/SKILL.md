# Phase 4：最窄切入点 (Narrowest Wedge) 确定

## 阶段目标

基于 [Phase 3 Synthesis](../ux-phase3-synthesis/SKILL.md) 的仲裁结论，**聚焦真正重要的功能**，找出最小可行且有人愿意立即付费的 MVP。

## 核心原则

> **先做真正重要的事，而不是因为资源不足而删减**  
> **MVP 是最小可行且有人愿意立即付费的版本**  
> **聚焦核心问题，而不是全部功能**

## 输入与输出

| 类型 | 文件 | 说明 |
|------|------|------|
| **输入 (必需)** | `03-synthesis.md` | Phase 3 产出，包含仲裁结论和核心发现 |
| **输入 (可选)** | `00-context.md` | Phase 0 产出，包含资源和约束信息 |
| **输出** | `04-mvp-scope.md` | MVP 范围定义：Must-have + Out of Scope + 成功指标 |

---

## 执行流程

### 步骤 1：读取并解析输入文件

**1.1** 读取 `03-synthesis.md`

如果文件不存在，提示用户先运行 `/ux:phase3-synthesis`。

从 synthesis.md 中提取以下关键信息：

- 最终版 Problem Statement（问题陈述）
- 优先 Persona（优先用户画像）
- 经过仲裁的核心发现（核心洞察）
- 进入 MVP Scoping 的明确边界（范围边界）

**1.2** 检查并读取 `00-context.md`（如存在）

如存在，提取以下关键约束作为**裁剪决策的参考**：

| 约束类型 | 说明 | 使用方式 |
|----------|------|----------|
| 工程资源 | 可用人数和人周估算 | 仅作参考，不决定功能保留与否 |
| 技术红线 | 不可改变的技术决定 | 与红线冲突的功能直接进入 Out of Scope |
| Deadline | 硬性时间节点 | 了解时间限制，优先保留核心功能 |
| 系统对接 | 必须对接的外部系统 | 影响功能实现复杂度的硬性集成依赖 |

**1.3** 如 `00-context.md` 不存在

使用 AskUserQuestion tool 向用户询问：

> "Phase 0 的现状信息不存在，我需要了解真实约束以便参考裁剪，请告诉我：
> 1. 可用工程师人数？
> 2. 目标上线时间？
> 3. 有没有技术上'绝对不能动'的东西？"

---

### 步骤 2：全量 Backlog 发散

根据 synthesis.md 的核心发现，列出**所有可能的功能点**。

**原则**：先发散，后收敛。不要在列举阶段做判断，将所有可能的功能点全部列出。

**输出**：功能点列表（待验证）

使用 AskUserQuestion tool 确认：

> "根据调研发现，我列出了以下可能的功能点。请确认是否有遗漏？
> [功能点列表]"

---

### 步骤 3：确定最窄切入点

用以下问题引导团队找到"真正重要"的功能（MVP）：

| 问题 | 目的 |
|------|------|
| 如果只解决一个最迫切需要验证的策略，这个 MVP 多久能完成？ | 确定 MVP 范围的时间可行性 |
| 用户最迫切想验证的功能是什么？ | 识别核心需求 |
| 有哪些功能能让用户立即付费试用？ | 识别价值锚点 |

> **重要提示**：资源限制仅作为参考，不因为资源少就删掉重要功能，而是确保团队能调动资源完成真正重要的 MVP。

---

### 步骤 4：反向验证（砍刀测试）

对每个保留的功能逐一进行以下测试：

**测试问题 1**：不做这个功能，用户还能用吗？

- → 能用 → **砍掉**

**测试问题 2**：这是解决问题还是让产品看起来专业？

- → 后者（仅提升专业感）→ **砍掉**

**测试问题 3**：这个功能是服务优先 Persona 的核心痛点，还是服务"也许有用"的场景？

- → 后者 → **砍掉**

**如有争议**：使用 AskUserQuestion tool 让用户最终决定。

---

### 步骤 5：定义成功指标

为 MVP 定义 **2-3 个可量化、可测量** 的 KPI。

使用以下问题引导：

> "MVP 上线后，怎么判断它成功了？请给出：
> 1. 当前基线（现在是多少）
> 2. MVP 目标（上线 [N] 周后达到多少）
> 3. 测量方式（怎么测）"
