## 1. 系统概述

通过连续阶段将模糊的业务需求转化为可落地的 PRD 文档。

### 核心目标

- **自动驾驶**，确定性高的环节由 AI 自动完成，人类负责 review 和决策
- **容易上手**，先定大方向，持续补充和优化规则细节，逐步引导用户去完成设计产品，并提供易用性检查
- **流程高效**，避免过度设计和文档冗余

### 系统组成

| 组件 | 数量 | 说明 |
|-----|------|------|
| Phase Skills | 12 个 | Phase 0-11 设计流程 |
| 辅助工具 | 1 个 | 文档一致性检查 |
| 依赖 Skill | 1 个 | 原型提示词生成器 |

## 3. 阶段详解

### Phase 0: 快速问题锚定

**目标**：半天内对齐"够用"的问题方向，输出草稿 Problem Statement

**核心原则**：
- 不要问"你需不需要这个功能"，要问"你现在遇到什么问题"
- 防方案跳跃：有人提方案，立刻引导回问题描述

**输入**：团队成员对核心痛点的描述

**输出**：
- `00-quick-anchor.md` - 草稿 Problem Statement + 争议点 + 调研方向
- `00-context-gaps.md` - 上下文缺口清单

**关键产出**：
| 输出项 | 说明 |
|-------|------|
| 草稿 PS | "[谁] 在 [场景] 遇到了 [困难]，导致 [影响]。现在的处理方法：[妥协方案]" |
| 争议点 | 至少识别 2-3 个需要并行调研验证的争议点 |
| 缺口清单 | 术语定义、流程背景、技术栈、组织结构、现有方案、约束条件 |

**够用的标准**：
- ✅ 大家对"我们在解决谁的问题"有基本共识
- ✅ 大家对"核心痛点大概是什么"有基本共识
- ✅ 已识别出 2-3 个需要并行调研验证的争议点

---

### Phase 1: 上下文补充收集

**目标**：基于缺口清单有针对性地收集信息，建立共同认知基准

**核心原则**：只记录"现在是什么"，不讨论"应该做什么"

**输入**：
- `00-quick-anchor.md`
- `00-context-gaps.md`
- 现有背景文档（可选）

**输出**：`00-context.md`

**收集维度**：

| 缺口类型 | 收集内容 |
|---------|---------|
| 术语定义 | PS 中的专业术语含义 |
| 流程背景 | 问题发生的前置流程 |
| 技术栈 | 影响方案设计的技术信息 |
| 组织结构 | 决策链路和利益相关方 |
| 现有方案 | 当前的临时解决方案 |
| 约束条件 | 硬性限制（时间/预算/技术/合规） |

---

### Phase 2: 解决方案研究

**目标**：编写用户故事，调研业界解决方案

**执行策略**：两条工作流串行

```
Workstream 1: 用户故事编写 → Workstream 2: 业界方案调研
```

**用户故事三要素**：
| 要素 | 说明 | 示例 |
|-----|------|------|
| 角色（Who） | 谁要用这个功能 | 作为一个系统管理员 |
| 活动（What） | 要完成什么活动 | 我想要一键同步所有数据中心的 Redis 配置 |
| 价值（Value） | 带来什么价值 | 以便于减少重复配置工作，降低配置错误风险 |

**三类替代方案扫描**：
1. **开源方案** - GitHub 项目、开源工具/框架
2. **商业产品/SaaS** - 可直接采购的解决方案
3. **用户的"笨办法"** - Excel、Wiki、口头传递（这是最重要的竞品！）

**输入**：`00-quick-anchor.md`, `00-context.md`（可选）

**输出**：`02-solution-research.md`

---

### Phase 3: 多 Agent 对抗式 Synthesis（核心环节）

**目标**：通过 Agent Teams 真实并行对抗，输出经得起推敲的 Synthesis

**核心设计**：三角张力对抗

```
        🔴 Tim Cook（做减法）
           /\
          /  \
         /    \
        /  对抗 \
       /________\
🟡 Jonathan Ive    🟢 Craig Federighi
（做加法）          （定边界）
```

| 角色 | 使命 | 典型攻击 |
|-----|------|---------|
| **Tim Cook** | 质疑需求，为 MVP 做减法 | "你们只访谈了 3 个人，这代表所有用户吗？" |
| **Jonathan Ive** | 代表用户声音，为 MVP 做加法 | "如果只能解决一件事，我最想要的是..." |
| **Craig Federighi** | 施加约束，定义可行边界 | "这个需求需要重构核心架构，MVP 阶段不可行" |

**对抗规则**：
- 最少完成 **2 轮对抗** 后才能结束
- 矛盾必须原样暴露，不得调和
- 最终仲裁者必须明确选边站，不允许"都有道理"

**输入**：`00-quick-anchor.md`, `00-context.md`, `02-solution-research.md`

**输出**：`03-synthesis.md`（完整辩论记录 + 仲裁结论）

**关键产出**：
- 最终版 Problem Statement
- 优先 Persona（MVP 阶段服务谁）
- 经过仲裁的核心发现
- 进入 MVP 的明确边界

---

### Phase 4: MVP 划定

**目标**：极度克制地做减法，只留 Must-have

**裁剪刀铁律**：
> **犹豫不决的，默认砍掉**  
> **"有了更好"，全部砍掉**  
> **通过资源限制假设逼出选择，不允许说"都要"**

**四个裁剪问题**：

| 问题 | 目的 |
|-----|------|
| 如果只有两周时间，你选什么？ | 时间限制 |
| 如果只有 1 个工程师，你选什么？ | 人力限制 |
| 技术上哪些现在不可行？ | 可行性过滤 |
| 优先 Persona 最先需要哪 1 件事？ | 用户价值排序 |

**砍刀测试**：
- 不做这个，用户还能用吗？ → 能 → **砍掉**
- 这是解决问题还是让产品看起来专业？ → 后者 → **砍掉**
- 这是服务核心痛点还是"也许有用"的场景？ → 后者 → **砍掉**

**输入**：`03-synthesis.md`, `00-context.md`（可选）

**输出**：`04-mvp-scope.md`

**关键产出**：
- ✅ Must-have（MVP 范围内）
- ❌ Out of Scope（本次不做，附排除理由）
- 📊 成功指标（可量化、可测量）

---

### Phase 5: 用户旅程设计

**目标**：为每个角色的每个场景绘制完整的用户旅程，明确用户在系统中的移动路径、触点和目标

**核心产出**：

**Part 1：完整用户旅程**
- 每个 Persona × 每个核心场景 = 独立用户旅程
- 包含要素：步骤、位置、动作、跳转、目标

**Part 2：功能模块结构图**
- 展示系统各 App 的功能模块父子关系（树状结构）
- 不包括业务流程和业务规则

**输入**：`04-mvp-scope.md`（必须），`03-synthesis.md`（可选）

**输出**：`05-user-journey.md`

---

### Phase 6: 流程演示 Demo 生成

**目标**：基于用户旅程生成可交互的流程演示 Demo，验证产品概念和核心流程

**核心价值**：
- 可视化呈现产品概念，提前发现流程问题
- 为项目章程和后续设计提供直观参考
- 便于 stakeholders 快速对齐产品方向

**Demo 类型**：

| 类型 | 适用场景 | 输出格式 |
|-----|---------|---------|
| **流程图 Demo** | 复杂业务流程展示 | Mermaid 流程图 |
| **线性流程 Demo** | 步骤清晰的场景 | HTML 原型 |
| **状态机 Demo** | 多状态流转场景 | 状态流转图 + 交互说明 |

**生成策略**：
1. 优先生成核心用户旅程的流程 Demo
2. 复杂分支场景使用条件分支图展示
3. 交互细节用文字说明，不追求高保真

**输入**：`05-user-journey.md`

**输出**：`06-flow-demo.html`（可交互 HTML 原型）

---

### Phase 7: 生成项目章程（Project Goal）

**目标**：将用户旅程和 MVP 范围转化为可执行的项目章程

**核心设计：渐进式规则策略**

```
Phase 6 Project Goal
├── 简单功能点 → 输出完整规则
└── 复杂功能点 → 标记 🔍（留待 Phase 7 详细设计）
```

**简单 vs 复杂判断标准**：

| 简单功能点（Phase 6 完整规则） | 复杂功能点（Phase 7 详细设计 🔍） |
|---------------------------|---------------------------|
| 业务逻辑单一，无复杂分支 | 涉及多状态、多分支 |
| 状态少（≤2 种） | 状态机（≥3 种状态） |
| 不涉及复杂异常处理 | 需要定义异常处理流程 |
| 数据格式和校验规则明确 | 涉及多系统数据交互 |
| 不涉及复杂权限判断 | 有复杂的权限判断逻辑 |

**输入**：`05-user-journey.md`, `04-mvp-scope.md`, `03-synthesis.md`（可选）

**输出**：`06-project-goal.md`

**项目章程结构**：
1. 识别角色（干系人及预期）
2. 项目背景和目标
3. 需求与功能点（含渐进式规则）
4. 功能点复杂度汇总
5. 整体信息架构总览
6. 辅助信息（术语表、Out of Scope）
7. 项目成功指标
8. 下一步计划

**检查闭环**：
```
初稿 → 用户检查 → 修正 → 确认通过
```

### Phase 7: 复杂功能点规则详细设计

**目标**：针对 Phase 6 标记 🔍 的复杂功能点，进行详细规则拆解

**核心原则**：不产生新文档，直接更新 `06-project-goal.md`

**五种规则模板**：

| 模板类型 | 适用场景 | 包含内容 |
|---------|---------|---------|
| **状态机类** | 涉及多个状态和流转 | 状态定义、流转图、流转规则、异常处理 |
| **流程类** | 多步骤流程 | 流程图、步骤详情、分支判断、取消/回退 |

| **集成类** | 多个系统数据交互 | 数据映射、调用流程、失败处理、幂等性 |
| **权限类** | 复杂权限判断 | 角色定义、权限矩阵、数据权限 |

**更新步骤**：
1. 确认功能点类型
2. 选择模板并收集信息
3. 输出规则初稿
4. 迭代修正
5. **更新 Project Goal**（添加「详细规则」章节，移除 🔍 标记）

**输入**：`06-project-goal.md`

**输出**：更新后的 `07-project-goal.md`

### Phase 10: 页面设计任务管理

**目标**：按页面维度拆解设计任务，引导用户使用外部 AI 工具完成原型设计

**核心定位**：
- ✅ 设计任务拆解与管理
- ✅ 调用 generator 生成 AI 提示词
- ✅ 收集原型方案链接
- ❌ 不直接生成提示词（由 generator 负责）
- ❌ 不直接完成具体页面设计

**工作流程**：

```
步骤1: 拆解设计任务
    ↓ 用户 Review
步骤2: 推进设计任务（循环）
    ├── 调用 generator 生成提示词
    ├── 用户用外部 AI 工具生成原型
    └── 用户返回原型链接 → 标记完成
步骤3: 检查补充任务
    ↓ 有补充 → 回到步骤2
步骤4: 更新 Project Goal
    └── 补充原型方案链接
```

**依赖 Skill**：`ux-prototype-prompt-generator`

**推荐的 AI 工具**：
- **Figma Make** — 直接粘贴提示词生成设计
- **Claude/ChatGPT + Figma** — 细化设计后手动绘制
- **V0.dev** — 生成可交互原型

**输入**：`07-project-goal.md`

**输出**：
- `10-design-tasks.md` - 设计任务清单（追踪状态）
- 更新后的 `07-project-goal.md`（补充原型方案链接）

### Phase 11: PRD 生成

**目标**：整合项目章程中的背景、需求、规则和原型方案，生成面向研发和测试的 PRD

**核心原则**：
- ✅ **结果导向** — 只保留最终结论，去除调研/讨论过程
- ✅ **研发友好** — 提供足够背景、清晰需求、明确规则
- ✅ **测试友好** — 规则清晰，便于编写测试用例
- ❌ **无过程信息** — 不包含调研记录、讨论过程、决策理由

**去除的内容**：

| 类型 | 示例 |
|-----|------|
| 调研过程 | "通过 5 轮用户访谈发现..." |
| 讨论记录 | "Phase 3 中 Tim Cook 质疑..." |
| 复杂度标记 | "此功能点标记为 🔍" |
| 决策过程 | "选择方案 A 而非方案 B 是因为..." |
| 逐步推导 | "首先分析...然后得出..." |

**PRD 结构**：
1. 项目背景（概述、目标用户、目标、成功指标）
2. 功能需求（需求总览、详细功能点、业务规则、验收标准）
3. 页面结构（信息架构、页面清单、页面流转）
4. 非功能需求（性能、兼容性、安全）
5. 附录（术语表、变更记录）

**输入**：`07-project-goal.md`

**输出**：`11-prd.md`

## 4. 辅助工具

### ux-doc-consistency-check: 文档一致性检查

**定位**：独立于设计流程的维护工具

**使用场景**：
1. **更新 Context 后** — 检查对下游文档的影响
2. **任意阶段完成后** — 验证产出与上游文档一致
3. **定期维护检查** — 全面扫描所有文档
4. **发现矛盾时** — 定位问题源头

**检查维度**：

| 维度 | 检查内容 |
|-----|---------|
| **版本一致性** | 文档引用是否为最新 |
| **信息一致性** | 技术栈、约束、角色、功能范围是否一致 |
| **影响完整性** | 上游变更是否反映到下游 |
| **逻辑自洽性** | Problem → MVP → 功能点 → 验收标准 是否闭环 |

**检查模式**：

| 模式 | 适用场景 |
|-----|---------|
| **全面检查** | 定期执行，扫描所有文档 |
| **变更检查** | 更新后执行，基于最近变更 |
| **特定检查** | 检查指定文档之间的特定信息 |

**输出**：`consistency-check-report-{日期}.md`

### ux-prototype-prompt-generator: 原型提示词生成器

**定位**：为 Figma Make 等 AI 工具生成简洁的业务逻辑提示词

**核心设计**：只保留 AI 无法从标准源获取的信息

| ✅ 保留 | ❌ 移除 |
|--------|--------|
| 数据结构（字段、类型、规则） | 页面整体布局（→ 低保真原型） |
| 状态流转（什么条件下变为什么状态） | 视觉风格（→ UIKit） |
| 权限逻辑（什么角色看到什么） | 标准交互（→ 组件库） |
| 计算规则（派生数据如何计算） | 重复的业务逻辑描述 |

**输出格式**：

```markdown
🎯 设计范围（3句话）

🧩 业务上下文
【目标用户】
【解决的问题】

📋 数据与规则
【核心数据结构】
【状态定义】（如适用）
【权限规则】（如适用）
【计算/派生规则】（如适用）
```

## 6. 设计原则

### 6.1 流程设计原则

| 原则 | 说明 |
|-----|------|
| **渐进式规则** | 不简单统一标准，而是根据复杂度分层处理 |
| **单一数据源** | Phase 6 直接更新 Project Goal，不新建文档 |
| **可追溯** | 每个功能点都能追溯到上游文档 |
| **检查闭环** | 每个阶段都有检查确认环节 |
| **文档分离** | 不同受众的文档分离（章程 vs 设计任务 vs PRD）|
| **模板驱动** | 复杂场景提供模板，确保完整性 |

### 6.2 Phase 0-4 调研阶段原则

- **防方案跳跃**：有人提方案，立刻引导回问题描述
- **够用即出发**：不完美但够用的草稿 PS 即可进入下一阶段
- **矛盾要暴露**：不要调和不同意见，原样记录
- **资源限制假设**：用"两周限制"、"单工程师"等假设逼出真实优先级

### 6.3 Phase 5-11 设计阶段原则

- **渐进式规则**：简单功能点完整输出，复杂功能点标记 🔍 留待 Phase 8
- **不产生技术细节**：项目章程说「做什么」，不说「怎么做」
- **保持单一数据源**：Phase 8 直接更新 Project Goal
- **结果导向**：PRD 只保留最终结果，去除过程信息

### 6.4 文档设计原则

- **最小修改原则**：更新文档时优先最小修改，保持原有结构稳定
- **显式标记**：不确定的信息必须标注「待确认」，不允许猜测
- **可追溯**：每个需求/功能点都能追溯到上游文档
- **检查闭环**：每个阶段都有检查确认环节
