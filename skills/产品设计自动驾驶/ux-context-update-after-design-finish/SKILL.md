# Context Update After Design 产品上下文更新器

在产品设计流程完成后，根据原始 context.md 和本次产出的 prd.md，生成更新后的产品上下文文档。

## When to use

- ✅ 完成一个功能迭代的产品设计后，需要更新产品上下文
- ✅ 设计流程结束，产出 PRD 文档，需要同步更新上下文
- ✅ 需要将设计结果转化为下一轮设计的输入时

## When NOT to use

- ❌ 设计流程尚未完成，没有 PRD 产出
- ❌ 只是需要查看当前上下文，不需要更新
- ❌ 需要保留设计过程信息（应使用完整设计文档）

---

## Core Principle

**结果导向，只记录最终状态**：

| 维度 | 说明 |
|------|------|
| ✅ **输入匹配** | 基于原有的 context.md 和新的 prd.md |
| ✅ **状态记录** | 带日期的 context 文件（如 context-2026-04-07.md）只包含更新后的最终状态 |
| ✅ **功能模块化** | 每个顶层功能模块独立章节 |
| ✅ **规则提取** | 记录状态定义、权限规则等业务规则 |
| ❌ **无过程信息** | 不包含设计讨论、决策过程、调研记录 |
| ❌ **无变更记录** | 不显示"什么发生了变化"这类对比信息 |
| ❌ **无项目背景** | 不包含项目背景描述 |

---

## Input

### Required

- `context.md` - 设计前的产品上下文文档（现有功能模块、用户场景、技术栈等）
- `prd.md` - 本次设计产出的产品需求文档

### Optional

- 如果 context.md 不存在，提示用户提供或创建初始版本
- 如果 prd.md 不存在，提示用户先完成设计流程

---

## Output

`context-YYYY-MM-DD.md` - 带日期的产品上下文文档（日期为生成当天）

### 输出文件名格式

文件名使用 `context-YYYY-MM-DD.md` 格式，其中：
- `YYYY` = 4位年份
- `MM` = 2位月份（01-12）
- `DD` = 2位日期（01-31）

**示例**：
- `context-2026-04-07.md`
- `context-2026-05-01.md`

**优势**：
- 文件名直接显示日期，用户一眼识别版本
- 每次迭代生成新文件，无需手动修改文件名
- 用户可使用最新日期的文件开始下一轮设计

## Workflow

### Step 1：验证输入

1. 检查 `context.md` 是否存在
   - 如果不存在，创建初始版本（从 PRD 提取信息）

2. 检查 `prd.md` 是否存在
   - 如果不存在，提示用户先完成产品设计流程

### Step 2：读取并分析 context.md

读取原始上下文文档，提取：

- 信息架构图（现有模块结构）
- 功能模块（顶层功能模块）
- 用户场景（角色、职责、诉求）
- 技术栈
- 约束条件

### Step 3：读取并分析 prd.md

从 PRD 文档中提取：

1. **功能模块结构** - 识别顶层功能模块（如"变更管理系统"、"安全漏洞管理系统"）
2. **页面结构图** - 提取所有页面的结构图（来自 05-user-journey.md Part 3 的内容）
3. **用户场景** - 提取角色定义、描述、职责、核心诉求
4. **功能点** - 提取功能点名称和描述
5. **状态定义** - 提取状态机定义（状态、进入条件、离开条件、页面表现）
6. **权限规则** - 提取角色可见性和可操作权限
7. **业务规则** - 提取其他业务规则
8. **技术要求** - 提取技术栈相关要求
9. **约束条件** - 提取非功能需求中的约束

### Step 4：合并生成带日期的上下文文档

文件名格式：`context-YYYY-MM-DD.md`（使用当前日期）

将原始 context.md 的基础信息与 PRD 中提取的新信息合并：

1. **更新信息架构图** - 反映新的模块结构（不标注新增模块）
2. **提取页面结构图** - 从 PRD 或 05-user-journey.md 提取所有页面结构图
3. **按功能模块组织** - 每个顶层功能模块一个大章节
4. **提取用户场景** - 按角色记录描述、职责、核心诉求
5. **提取功能点规则** - 记录状态定义、权限规则、业务规则
6. **保留其他信息** - 技术栈、约束条件（如有更新则覆盖）

### Step 5：验证输出

确认带日期的上下文文档（如 `context-2026-04-07.md`）包含：

- [ ] 目录（便于导航）
- [ ] 信息架构图（更新后）
- [ ] 页面结构图（本次迭代涉及的所有页面）
- [ ] 功能模块章节（每个顶层模块独立章节）
- [ ] 用户场景（角色、描述、职责、核心诉求）
- [ ] 功能点规则（状态定义、权限规则、业务规则）
- [ ] 技术栈（保留或更新）
- [ ] 约束条件（保留或更新）

确认带日期的上下文文档（如 `context-YYYY-MM-DD.md`）不包含：

- [ ] 项目背景
- [ ] 变更记录
- [ ] 设计过程信息

## Extraction Rules

### 从 context.md 提取（保留或覆盖）

| 内容 | 处理 |
|------|------|
| 信息架构图 | 保留，用 PRD 新模块结构更新 |
| 功能模块 | 用 PRD 中的新模块覆盖 |
| 用户场景 | 保留，合并 PRD 中的新角色 |
| 技术栈 | 保留，合并 PRD 中的新技术 |
| 约束条件 | 保留，合并 PRD 中的新约束 |

### 从 prd.md 提取

| 内容 | 来源章节 | 提取方式 |
|------|----------|----------|
| 功能模块 | PRD 第 2 章 "功能需求" | 识别顶层功能模块 |
| 页面结构图 | 05-user-journey.md Part 3 / Project Goal 第 4 章 | 提取页面布局和组件层级 |
| 用户场景 | PRD 第 1 章 "目标用户" | 提取角色+职责+诉求 |
| 功能点 | PRD 第 2 章 "功能需求" | 提取功能点名称和描述 |
| 状态定义 | PRD 功能点下的"状态定义"表格 | 提取状态机 |
| 权限规则 | PRD 功能点下的"权限规则"表格 | 提取角色权限 |
| 业务规则 | PRD 功能点下的"业务规则"表格 | 提取规则项 |
| 技术要求 | PRD 第 4 章 "非功能需求" | 提取技术栈相关 |
| 约束条件 | PRD 第 4 章 "非功能需求" | 提取约束 |

---

## 重要原则

### 只记录结果状态，不记录过程

| 应该包含 | 不应该包含 |
|----------|------------|
| 最终的功能模块结构 | 设计过程中的讨论 |
| 最终的页面结构图 | 页面结构变更历史 |
| 最终的用户场景 | 调研发现和分析 |
| 最终的规则（状态、权限、业务） | 方案对比和决策理由 |
| 当前的技术栈 | 哪些是"新增"的说明 |
| 当前的约束条件 | 变更记录 |

### 无缝衔接

- 带日期的 context 文件（如 `context-2026-04-07.md`）作为下次设计的输入
- 用户可直接使用最新版本的文件开始下一轮设计
- 不需要额外的转换或处理

### 信息架构图原则

- 展示系统完整的模块结构
- 不标注哪些是"新增"模块
- 保持层级清晰

## Troubleshooting

| 问题 | 解决方案 |
|------|----------|
| context.md 不存在 | 从 PRD 提取信息创建初始版本 |
| prd.md 不存在 | 提示用户先完成产品设计流程 |
| PRD 格式非标准 | 尝试识别关键章节，提取可用信息，标注待确认项 |
| 提取信息不完整 | 用"待确认"标注缺失项，提示用户补充 |
| 功能模块层级不清晰 | 与用户确认顶层模块划分 |
