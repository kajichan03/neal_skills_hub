# Phase 5：用户旅程设计（User Journey Mapping）

## 阶段目标

为每个角色的每个场景绘制完整的用户旅程，明确用户在系统中的移动路径、触点和目标，为后续详细功能设计提供完整的流程蓝图。

## 核心原则

> **从用户目标倒推** - 先明确用户最终要完成什么，再拆解中间步骤  
> **路径唯一性** - 每个步骤只有一个明确的下一步  
> **覆盖全流程** - 从进入系统到离开的完整路径，包括异常分支  
> **与 MVP 对应** - 每个步骤必须能在 MVP 功能清单中找到对应项

## 输入与输出

| 类型 | 文件 | 说明 |
|------|------|------|
| **输入 (必需)** | `04-mvp-scope.md` | Phase 4 产出，包含 MVP 范围内的功能清单 |
| **输入 (可选)** | `03-synthesis.md` | Phase 3 产出，包含 Persona 和核心场景 |
| **输入 (可选)** | `00-context.md` | Phase 0 产出，包含系统模块结构图 |
| **输出** | `05-user-journey.md` | 用户旅程 + 功能模块结构图 |

### 步骤 2：确定角色与场景矩阵

基于 synthesis.md 中的 Persona 和 MVP 范围内的功能，确定需要绘制的用户旅程。

使用 **AskUserQuestion tool** 确认：

> "根据 MVP 范围，我识别到以下角色和场景组合，请确认是否完整？
> 
> [角色] × [场景] 矩阵"

---

### 步骤 3：绘制用户旅程

对于每个 Persona × 场景组合，按以下结构绘制：

**格式要求**：

```markdown
## [Persona 名称] - [场景名称]

| Step | 位置 | 动作 | 跳转 | 目标 |
|------|------|------|------|------|
| 1 | [模块/子模块/页面] | [具体操作] | [下一步流向] | [用户目标] |
| 2 | | | | |
| ... | | | | |
```

**要素说明**：

| 要素 | 说明 |
|------|------|
| 步骤（Step） | 步骤序号，从 1 开始，连续编号 |
| 位置（Location） | 所在模块 / 子模块 / 页面，三级精确 |
| 动作（Action） | 用户在此步骤执行的具体操作 |
| 跳转（Jump） | 此步骤之后流向哪里（下一页 / 弹窗 / 跳转 App / 流程结束） |
| 目标（Goal） | 用户完成此步骤要达成的子目标 |

---

### 步骤 4：标注关键触点

在用户旅程中标注以下关键信息：

| 标记 | 类型 | 说明 |
|------|------|------|
| ⭐ | 核心触点 | 用户完成主要目标的关键页面 |
| 🔀 | 决策点 | 用户需要做选择的节点 |
| ⏳ | 等待点 | 系统处理中用户等待的环节 |
| 🚪 | 跳出点 | 用户离开当前流程的节点 |

---

### 步骤 5：识别系统模块

从以下来源提取所有涉及的 App 和模块：
- `00-context.md` 中的已有系统模块结构图（如存在）

### 步骤 6：构建功能模块结构图

使用 Unicode 树状结构展示父子关系，支持标注新增/修改状态：

**结构符号规范**：
- `└──` 表示最后一个子项（无后续兄弟节点）
- `├──` 表示非最后子项（有后续兄弟节点）
- `│   ` 用于维持树状对齐

**层级类型**：

| 层级类型 | 示例格式 | 说明 |
|----------|----------|------|
| App | `### 变更管理系统` | 顶级系统入口 |
| 模块 | `└── 发布单列表` | App 下的功能模块 |
| 页面 | `    └── 发布单详情页` | 模块下的具体页面 |
| Tab/区域 | `        ├── 基本信息 Tab` | 页面内的标签页或区域 |
| 子区域 | `            ├── 附件上传区域（顶部）` | 区域内的子区域 |
| 组件 | `                ├── 上传组件` | 最底层的具体组件 |

**状态标注**：
- `（新增）` - 新增功能模块
- `（修改）` - 有变更的功能模块
- `（删除）` - 已废弃功能模块（仅在重构场景使用）

**示例**：

```markdown
### 变更管理系统
└── 发布单列表
    └── 发布单详情页
        ├── 基本信息 Tab
        ├── 变更内容 Tab
        └── 安全漏洞情况 Tab（新增）
            ├── 附件上传区域（顶部）
            │   ├── 上传组件
            │   └── 附件列表
            └── 漏洞列表区域（底部）
                ├── 代码漏洞列表
                ├── 镜像漏洞列表
                ├── 组件漏洞列表
                └── 操作按钮（已修复重扫）

### 安全漏洞管理系统
└── 漏洞列表
    └── 批量漏洞延期修复弹窗（修改）
        ├── 原有内容（漏洞信息、延期时间选择等）
        └── 新增 Section：延期修复审批记录
            └── 表格：发布单名称 | 发布时间 | 系统 | 审批记录
```

**关联用户旅程标注方式**：
- App 级别：在 App 名称后标注 `← 对应旅程：[Persona]-[场景]`
- 模块级别：在模块后标注 `← Step X`
- 页面级别：在页面后标注 `← Step X-Y`

**重要约束**：
- 只列出功能模块名称，不包含业务流程
- 不包含业务规则描述
- 层级反映模块的包含关系

### 步骤 7：合并与校验

执行以下校验：

| 校验项 | 说明 |
|--------|------|
| 完整性 | 每个 MVP 功能都出现在用户旅程中 |
| 一致性 | 跳转关系逻辑自洽 |
| 可达性 | 用户能通过有限步骤完成目标 |

## 护栏

- **每个角色-场景组合独立** - 不允许合并，必须单独绘制
- **四要素完整** - 位置、动作、跳转、目标缺一不可
- **功能模块不含业务流程** - 只展示父子层级结构
- **跳转逻辑自洽** - 不能出现"死胡同"或无限循环
- **与 MVP 对应** - 用户旅程中的每个步骤都能在 MVP 功能清单中找到对应项
