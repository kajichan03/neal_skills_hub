# Project Engineering - Checklist

> 软件工程项目管理检查清单

---

## 项目创建检查

### 文档结构检查

```
projects/{name}/
├── progress.md              ✓ 初始化当前状态
├── iteration-plan.md        ✓ 创建空文件（迭代开始时填充）
├── requirements.md          ✓ 创建空文件
├── architecture.md          ✓ 创建空文件
├── decisions.md             ✓ 创建空文件
├── INTERRUPT.md             ✓ 初始化模板
└── changes/                 ✓ 创建目录
    ├── requirements.md      ✓ 创建空文件
    └── architecture.md      ✓ 创建空文件
```

### INTERRUPT.md 初始化模板

```markdown
# 中断恢复检查

## 当前状态
- **项目**: {name}
- **当前迭代**: Iteration 1
- **整体进度**: 0%
- **最后更新**: YYYY-MM-DD

## 上次对话
- **目标**: 项目初始化
- **已完成**: 创建项目结构
- **阻塞**: 无

## 本次对话目标
- [ ] 确定迭代 1 目标
- [ ] 创建 iteration-plan.md
```

## 迭代规划检查

### 创建 iteration-plan.md 时

- [ ] 已明确迭代目标和范围
- [ ] 任务已拆解为可执行单元（建议单个任务 2-8 小时）
- [ ] 每个任务有唯一 ID（如 I1-001）
- [ ] 已估算工时
- [ ] 已标注任务依赖关系
- [ ] 已定义验收标准
- [ ] 已评估风险并记录缓解措施
- [ ] **【关键文档待 Review】** 已提示用户 Review iteration-plan.md

### iteration-plan.md Review 检查点

- [ ] 任务拆解是否合理（粒度适中）
- [ ] 工时估算是否可行
- [ ] 依赖关系是否清晰
- [ ] 验收标准是否可衡量

---

## 关键文档 Review 检查

### requirements.md Review（阻塞式）

**触发时机**：生成或更新后

Review 检查点：
- [ ] 需求是否覆盖所有原始诉求？
- [ ] 优先级划分是否合理（P0/P1/P2）？
- [ ] 验收标准是否可衡量、可验证？
- [ ] 非功能性需求是否完整（性能、安全、兼容）？

**通过后**：继续下一步
**不通过**：记录修改意见，返回修改

### architecture.md Review（阻塞式）

**触发时机**：生成或更新后

Review 检查点：
- [ ] 架构设计是否合理？
- [ ] 模块划分是否清晰？
- [ ] 接口定义是否完整？
- [ ] 技术选型是否说明理由？

**通过后**：继续下一步
**不通过**：记录修改意见，返回修改

### progress.md 首次拆分 Review（阻塞式）

**触发时机**：首次完成 Iteration 和 task 拆分后

Review 检查点：
- [ ] 迭代范围是否合理？
- [ ] 任务优先级是否正确？
- [ ] 资源分配是否可行？

**通过后**：继续下一步
**不通过**：记录修改意见，返回修改

## 日常开发检查

### 每轮对话开始

- [ ] 已阅读 INTERRUPT.md 了解中断状态
- [ ] 已阅读 progress.md 了解当前进度
- [ ] 已确认本次对话目标

### 每轮对话结束

#### 文档更新检查

| 场景 | 检查项 |
|------|--------|
| 完成开发任务 | [ ] progress.md 已更新任务状态<br>[ ] 执行记录已添加（遇到的问题、解决方案） |
| 新增/修改需求 | [ ] requirements.md 已更新<br>[ ] changes/requirements.md 已记录变更原因 |
| 修改架构 | [ ] architecture.md 已更新<br>[ ] changes/architecture.md 已记录变更原因 |
| 做出技术决策 | [ ] decisions.md 已记录决策内容、原因、影响 |
| 无特殊事件 | [ ] progress.md 已更新当前状态 |

#### INTERRUPT.md 更新检查

- [ ] 当前迭代已更新
- [ ] 整体进度已更新
- [ ] 本次对话已完成内容已记录
- [ ] 如有阻塞，阻塞原因已记录
- [ ] 下次对话目标已列出

#### 提醒机制检查

- [ ] 如有重要文档更新，已提示用户（不阻塞）
- [ ] 格式："【重要文档已更新】{文档名}，建议检查..."

---

## 迭代归档检查

### 迭代完成时

- [ ] iteration-plan.md 已标记为"🟢 已归档"
- [ ] 已记录实际完成时间
- [ ] 已记录与计划的偏差分析
- [ ] progress.md 已更新为下一迭代或项目完成

### 归档标记格式检查

```markdown
> **状态**: 🟢 已归档（YYYY-MM-DD 完成）  
> **原计划**: N 个任务，预计 M 天  
> **实际完成**: N 个任务，实际 X 天  
> **偏差分析**: 提前/延后原因...
```

---

## 项目完成检查

- [ ] 所有迭代已归档
- [ ] progress.md 状态已更新为"🟢 已完成"
- [ ] 最终架构已归档到 architecture.md
- [ ] 所有关键决策已记录到 decisions.md
- [ ] INTERRUPT.md 已标记项目完成

---

## 常见错误规避

| 错误 | 正确做法 |
|------|----------|
| 关键文档更新后继续开发 | **必须阻塞等待 Review** |
| iteration-plan.md 在迭代中频繁修改 | **归档前不修改**，变化记录到 progress.md |
| progress.md 复制任务清单 | **引用任务ID**，详情去 iteration-plan 查 |
| 需求/架构变更不留痕 | **必须记录到 changes/ 目录** |
| 跳过 INTERRUPT.md | **每轮开始读，每轮结束写** |
| 任务粒度太大（> 1天）或太小（< 30分钟） | 控制在 2-8 小时 |
