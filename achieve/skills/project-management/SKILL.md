# Project Management Skill

> 个人项目追踪和管理方法论

---

## 概述

本 Skill 提供一套项目管理体系，支持三种模式：
- **轻量模式**：适合 1-3 天完成的小型项目
- **工程模式**：适合需要长期维护的软件工程项目（完整文档体系）
- **产品设计模式**：适合产品需求梳理和原型设计项目

---

## 项目模式选择

创建项目时选择适用模式：

| 模式 | 适用场景 | 文档要求 |
|------|---------|---------|
| **lightweight** | 脚本、工具、实验性项目 | progress.md + decisions.md |
| **engineering** | 软件系统、长期维护项目 | 完整文档体系（见下文） |
| **product-design** | 产品需求梳理、原型设计 | 项目章程 + 进度追踪 + 决策记录 |

## 产品设计模式文档体系（product-design）

适用于产品需求梳理和原型设计项目的文档结构：

```
projects/{name}/
├── Project_Goal.md          # 项目章程（需求来源）- 用户维护
├── progress.md              # 动态运行仪表盘 - 每轮对话更新
├── decisions.md             # 关键决策记录
├── input_product_doc.md     # 现有产品文档（参考）
└── output_product_doc.md    # 更新后的产品文档（产出）
```

> **⚠️ 注意**: Project_Goal.md 由用户手工维护，AI Agent 严禁修改。

### 项目阶段

产品设计项目分为以下阶段：

#### 阶段 1：编写项目背景和目标

- **输出文档**：Project_Goal.md（由用户编写）
- **Progress.md 记录**：里程碑完成

#### 阶段 2：拆解需求和功能点

- **输出文档**：Project_Goal.md（由用户编写）
- **Progress.md 记录**：里程碑完成
- **任务拆分**：完成后生成两个任务列表
  - **需求文档任务列表**：以需求和功能点为任务单位
  - **原型页面设计任务列表**：以页面为任务单位

#### 阶段 3：编写需求文档

- **输出文档**：Project_Goal.md（由用户编写）
- **Progress.md 记录**：更新需求文档任务进度

#### 阶段 4：验证需求文档

- **动作**：提示生成简略的"整体信息架构总览"，检查结构是否正确
- **Progress.md 记录**：里程碑完成

#### 阶段 5：推进设计任务

**步骤**：
1. 调用 skill `prototype-prompt-generator`，生成当前设计任务的 Figma Make Prompt
2. 用户基于 Prompt 完成设计后，**必须提供设计方案链接**
3. **【检查点】收到交付物（设计方案链接）后，才能标记任务为"已完成"**
4. 更新 Progress.md（状态 + 交付物链接）
5. 同步记录到 Project_Goal.md（用户确认后更新）

**交付物检查规则（强制）**：

| 任务状态变更 | 交付物要求 | 操作 |
|--------------|-----------|------|
| 🚧 待设计 → 🔄 设计中 | 无 | 允许 |
| 🔄 设计中 → ✅ 已完成 | **必须有设计方案链接** | 无交付物则保持"设计中" |
| 任何状态 → ❌ 已取消 | 无 | 允许 |

**禁止**：将任务标记为"✅ 已完成"时，交付物为"待补充"或为空。

#### 阶段 6：汇总并更新产品文档

- **目标**：
  1. 每个功能有具体、清晰的产品规则
  2. 每个功能记录设计原因（用户故事）
- **输出文档**：output_product_doc.md
- **Progress.md 记录**：里程碑完成
- **用户 Review**：要求用户 review，记录 review 结果
- **确认机制**：如有不确定的地方，标记并向用户提问确认

#### 阶段 7：检查点 - 需求发生变更

如需求发生变更：
1. 更新 decisions.md
2. 需求文档任务状态变为"需求变更，待更新"
3. 原型页面设计任务状态变为"需求变更，待更新"
4. 提示要更新产品文档
5. 给出原型修改建议：把需求变更喂给 Figma Make，而不是重新生成

## 工程模式文档体系（engineering）

适用于软件工程项目的完整文档结构（**精简版 2.0**）：

```
projects/{name}/
├── progress.md              # 动态运行仪表盘（必须）- 只保留当前状态和近期执行记录
├── iteration-plan.md        # 静态设计文档（必须）- 详细规划，完成后归档
├── requirements.md          # 需求文档（必须）
├── architecture.md          # 架构设计文档（必须）
├── decisions.md             # 关键决策记录（必须）
├── INTERRUPT.md             # 中断恢复检查（必须）
└── changes/                 # 变更记录目录
    ├── requirements.md      # 需求变更记录
    └── architecture.md      # 架构变更记录
```

> **⚠️ 注意**: tasks.md 已合并到 progress.md，不再需要单独维护。

### 文档职责分工（重要）

| 文档 | 性质 | 内容 | 更新时机 |
|------|------|------|----------|
| **iteration-plan.md** | 静态设计 | 详细的任务拆解、工时估算、依赖关系、验收标准 | 规划阶段写详细，完成后**归档冻结** |
| **progress.md** | 动态运行 | 当前状态、最近完成（3-5条）、下一步、阻塞问题 | **每轮对话更新** |

**核心原则**：
1. **不重复**：progress 不复制 iteration-plan 的任务清单
2. **引用而非复制**：progress 中只写任务ID（如 I2-011），详情去 iteration-plan 查
3. **归档即冻结**：iteration-plan 在迭代完成后标记"已归档"，不再修改状态

### 迭代规划文档（iteration-plan.md）规范

**生命周期**：
```
1. 迭代开始时：创建详细规划（任务、工时、依赖、验收标准）
2. 迭代进行中：不修改（除归档时添加完成信息）
3. 迭代结束时：标记"已归档"，记录实际完成时间、与计划的偏差
```

**归档标记格式**：
```markdown
## Iteration X: XXX

> **状态**: 🟢 已归档（2026-XX-XX 完成）  
> **原计划**: N 个任务，预计 M 天  
> **实际完成**: N 个任务，实际 X 天  
> **偏差分析**: 提前/延后原因...
```

### 进度追踪文档（progress.md）规范

**内容结构**：
```markdown
## 📊 当前状态
- **当前迭代**: Iteration X
- **整体进度**: XX%
- **当前聚焦**: 正在进行的任务
- **阻塞问题**: 如有

## 📋 最近完成（倒序，只保留最近 3-5 条）
### 时间 - 任务ID 完成
- 简要描述关键成果
- 关键指标（测试数、覆盖率等）
- 🔗 详细规划见 iteration-plan.md#任务ID

## 🎯 下一步
- 下一个任务（引用 iteration-plan）
```

**更新原则**：
- 每次对话结束前更新
- 只记录"变化"，不重复记录"计划"
- 历史记录过多时，归档到"历史记录"章节或删除

### 文档更新触发条件（强制）

| 场景 | 必须更新的文档 | 更新内容 |
|------|---------------|----------|
| 新增功能需求 | `requirements.md` | 添加需求描述、优先级、验收标准 |
| 确定技术方案 | `architecture.md` | 更新架构设计、模块关系、接口定义 |
| 开始新迭代 | `iteration-plan.md` | 迭代目标、任务拆解、时间线 |
| 完成开发任务 | `tasks.md` | 任务执行记录、遇到的问题、解决方案 |
| 修改需求 | `changes/requirements.md` | 变更原因、影响分析、相关人通知 |
| 修改架构 | `changes/architecture.md` | 变更原因、兼容性说明、迁移方案 |
| 编写测试 | `test-cases.md` | 测试用例、测试数据、预期结果 |
| 每轮对话结束 | `progress.md` | 当前进度百分比、阻塞问题、下一步 |

> ⚠️ **重要**：工程模式项目，每次对话结束前必须检查并更新相关文档。

## 三层约束机制

### 第一层：Skill 规范（本文件）

明确声明工程模式需要维护哪些文档，以及何时更新。

### 第二层：CHECKLIST 强制检查

详见 `CHECKLIST.md`，包含：
- 项目创建检查清单（区分 lightweight/engineering/product-design）
- 文档更新检查清单
- 对话结束检查清单

### 第三层：INTERRUPT.md 恢复检查

工程模式项目必须包含 `INTERRUPT.md`，每次对话开始时自动加载，包含：
- 当前进度回顾
- 文档更新状态检查
- 本次对话目标确认

## 关键文档处理流程（engineering 模式强制）

### 处理原则

**任何时候**更新以下关键文档，必须**立即停止当前工作**，执行 Review 流程：

### 关键文档清单

| 文档 | 触发时机 | 阻塞要求 |
|------|----------|----------|
| `requirements.md` | 生成或更新后 | ⛔ **必须 Review 通过后才能继续** |
| `architecture.md` | 生成或更新后 | ⛔ **必须 Review 通过后才能继续** |
| `progress.md` | **首次**完成拆分 Iteration 和 task 后 | ⛔ **必须 Review 通过后才能继续** |

### Agent 行为规范（必须遵守）

**步骤 1：更新前声明**
```
我：【关键文档更新】准备更新 {文档名}，补充/修改 xxx 内容...
```

**步骤 2：执行更新**
- 更新文档内容
- 确保文档质量

**步骤 3：更新后阻塞声明（必须）**
```
我：【关键文档待 Review】{文档名} 已更新

本次更新摘要：
1. xxx
2. xxx

Review 检查点：
- [ ] 检查点 1
- [ ] 检查点 2

请回复 "Review 通过" 或提出修改意见。
⛔ **当前工作已暂停，等待 Review 确认**
```

**步骤 4：等待确认**
- 等待用户回复
- 如未通过：记录修改意见，返回步骤 2
- 如已通过：继续下一步工作

> ⚠️ **重要**：违反此流程（更新关键文档后继续开发而未 Review）属于严重违规。

## 关键文档 Review 机制

### 关键文档清单

以下文档在**生成或更新后**必须交付给用户 Review，**只有通过后才能继续下一步动作**：

| 文档 | 触发时机 | 阻塞进度 | Review 要点 |
|------|----------|----------|-------------|
| `requirements.md` | 生成或更新后 | ✅ 是 | 需求完整性、优先级合理性、验收标准可衡量 |
| `architecture.md` | 生成或更新后 | ✅ 是 | 架构合理性、模块划分清晰、接口定义完整 |
| `progress.md` | **首次**完成拆分 Iteration 和 task 后 | ✅ 是 | 任务拆解合理、工时估算可行、依赖关系清晰 |

### Review 流程

```
1. AI 完成关键文档编写/更新
   │
   ▼
2. AI 明确提示用户：
   "【关键文档待 Review】{文档名} 已更新，请确认："
   
   Review 检查点：
   - [ ] 检查点 1: xxx
   - [ ] 检查点 2: xxx
   - [ ] 检查点 3: xxx
   
   请回复 "Review 通过" 或提出修改意见。
   │
   ▼
3. 用户 Review
   ├─ 通过 → 继续下一步
   └─ 不通过 → 记录修改意见，返回步骤 1 修改
```

### 示例提示

**requirements.md 更新后**：

```markdown
【关键文档待 Review】requirements.md 已更新

本次更新内容：
- 新增 3 个 P0 需求（REQ-001/002/003）
- 定义了验收标准
- 补充了非功能性需求

Review 检查点：
- [ ] 需求是否覆盖所有原始诉求？
- [ ] 优先级划分是否合理？
- [ ] 验收标准是否可衡量、可验证？

请回复 "Review 通过" 或提出修改意见。
（此文档 Review 阻塞后续任务）
```

## 重要文档检查提醒（engineering 模式）

### 重要文档清单

以下文档在**更新后**提醒用户检查，**不阻塞进度**，只提醒：

| 文档 | 触发时机 | 阻塞进度 | 提醒内容 |
|------|----------|----------|----------|
| `progress.md` | **每次**更新后 | ❌ 否 | 本次更新摘要、下一步行动 |
| `decisions.md` | **每次**决策后 | ❌ 否 | 决策内容、原因、影响 |

### 提醒流程

```
1. AI 完成重要文档更新
   │
   ▼
2. AI 提示用户（不等待确认）：
   "【重要文档已更新】{文档名}，建议检查："
   
   本次更新摘要：
   - 要点 1: xxx
   - 要点 2: xxx
   - 要点 3: xxx
   
   不阻塞进度，您可在方便时 Review。
   │
   ▼
3. 继续下一步动作
```

### 示例提示

**progress.md 更新后**：

```markdown
【重要文档已更新】progress.md

本次更新摘要：
- 整体进度: 27% → 36%
- 完成 TASK-004: 制定测试策略
- Iteration 1 已完成 (100%)
- 下一步: 开始 Iteration 2

不阻塞进度，您可在方便时 Review。
```

**decisions.md 更新后**：

```markdown
【重要文档已更新】decisions.md

本次新增决策：
- 决策 1: Skill 架构采用基类 + 注册表模式
  - 原因: 类型安全、易于测试、支持版本管理
- 决策 2: Token 管理采用环境变量 + 本地文件混合模式
  - 原因: 平衡安全性和便利性

不阻塞进度，您可在方便时 Review。
```

## 使用流程

### 1. 创建新项目

```bash
python3 skills/project-management/tools/create-project.py
# 选择对应模式
```

### 2. 日常协作流程（产品设计模式）

**每次对话开始**：
```
1. 阅读 progress.md 了解当前阶段和进度
2. 阅读 Project_Goal.md 了解项目章程（只读）
3. 阅读 decisions.md 了解历史决策
```

**每次对话结束**：
```
1. 更新 progress.md 记录当前阶段进度
2. 如有决策，更新 decisions.md
3. 如需用户确认，明确提示 Review
```

### 3. 日常协作流程（工程模式）

**每次对话开始**：
```
1. 阅读 INTERRUPT.md 了解中断状态
2. 阅读 progress.md 了解当前进度
3. 阅读相关需求/设计文档
```

**每次对话结束**：
```
1. 更新 progress.md 记录进度
2. 检查是否需要更新其他文档（对照触发条件表）
3. 更新 INTERRUPT.md 记录当前状态
```

### 4. 项目完成

1. 更新 progress.md 状态为 "🟢 已完成"
2. 确保所有文档已归档
3. 更新 TODO.md 标记完成
4. 移动到 ARCHIVE/ (可选)

## 最佳实践

### 产品设计模式特有

1. **用户维护章程**：Project_Goal.md 由用户编写，AI 只读取
2. **双任务列表**：需求任务和原型任务并行跟踪
3. **设计交付物**：每个设计任务必须有设计方案链接
4. **变更影响追踪**：需求变更需同步更新所有相关任务状态

### 工程模式特有

1. **文档先行**：任何功能开发前，先更新需求/设计文档
2. **变更留痕**：需求或架构变更必须记录原因和影响
3. **测试同步**：功能开发与测试用例编写同步进行
4. **定期审计**：每周运行 check-structure.py 检查文档健康度

### 通用

1. **进度透明**: 每次会话更新 progress.md
2. **阻塞即记录**: 遇到问题立即记录阻塞原因
3. **决策留痕**: 关键决策写入 decisions.md
4. **小步快跑**: 项目粒度控制在合理范围内
