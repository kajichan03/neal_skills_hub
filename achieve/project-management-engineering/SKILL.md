# Project Engineering Skill

> 软件工程项目迭代开发管理方法论

---

## 概述

本 Skill 提供软件工程项目的完整管理体系，适用于：
- 需要长期维护的软件系统
- 多迭代周期的开发任务
- 团队协作的工程项目

---

## 文档体系

```
projects/{name}/
├── progress.md              # 动态运行仪表盘（必须）- 每轮对话更新
├── iteration-plan.md        # 静态设计文档（必须）- 详细规划，完成后归档
├── requirements.md          # 需求文档（必须）
├── architecture.md          # 架构设计文档（必须）
├── decisions.md             # 关键决策记录（必须）
├── INTERRUPT.md             # 中断恢复检查（必须）
└── changes/                 # 变更记录目录
    ├── requirements.md      # 需求变更记录
    └── architecture.md      # 架构变更记录
```

## 触发条件

**当以下情况时，必须使用本 Skill：**

- 用户提到"迭代"、"Iteration"、"进度"、"任务"
- 用户说"开始新迭代"、"创建迭代计划"
- 用户更新任务状态（如"I1-001 完成了"）
- 用户询问"进度如何"、"状态怎么样"
- 用户提到"阻塞"、"blocker"、"遇到问题"
- 用户说"归档"、"archive"迭代
- 用户说"继续"、"恢复"之前的工作
- 用户提到"Review"、"评审"文档
- **对话开始时，检查项目文档完整性**

**关键词：** 迭代、进度、任务、I1-xxx、阻塞、归档、继续、Review

## 文档职责分工

| 文档 | 性质 | 内容 | 更新时机 |
|------|------|------|----------|
| **iteration-plan.md** | 静态设计 | 详细任务拆解、工时估算、依赖关系、验收标准 | 规划阶段写详细，完成后**归档冻结** |
| **progress.md** | 动态运行 | 当前状态、最近完成（3-5条）、下一步、阻塞问题 | **每轮对话更新** |

**核心原则**：
1. **不重复**：progress 不复制 iteration-plan 的任务清单
2. **引用而非复制**：progress 中只写任务ID（如 I2-011），详情去 iteration-plan 查
3. **归档即冻结**：iteration-plan 在迭代完成后标记"已归档"，不再修改

## 迭代生命周期

### 1. 迭代规划

**创建 iteration-plan.md**：
```markdown
## Iteration X: 迭代目标

> **状态**: 🟡 进行中  
> **时间**: YYYY-MM-DD 至 YYYY-MM-DD

### 任务列表
| ID | 任务 | 工时 | 依赖 | 验收标准 |
|----|------|------|------|----------|
| I{X}-001 | xxx | 2h | - | xxx |

### 风险评估
- 风险1: xxx，缓解措施: xxx
```

### 2. 迭代执行

**更新 progress.md**：
```markdown
## 📊 当前状态
- **当前迭代**: Iteration X
- **整体进度**: XX%
- **当前聚焦**: 任务ID - 任务名
- **阻塞问题**: 如有

## 📋 最近完成（倒序，只保留最近 3-5 条）
### YYYY-MM-DD - 任务ID 完成
- 关键成果: xxx
- 指标: 测试数、覆盖率等
- 🔗 详细规划见 iteration-plan.md#任务ID

## 🎯 下一步
- 下一任务（引用 iteration-plan）
```

### 3. 迭代归档

**完成时更新 iteration-plan.md**：
```markdown
## Iteration X: XXX

> **状态**: 🟢 已归档（YYYY-MM-DD 完成）  
> **原计划**: N 个任务，预计 M 天  
> **实际完成**: N 个任务，实际 X 天  
> **偏差分析**: 提前/延后原因...
```

## 每日协作流程

### 对话开始时
```
1. 阅读 INTERRUPT.md 了解中断状态
2. 阅读 progress.md 了解当前进度
3. 阅读相关需求/设计文档
```

### 对话结束时
```
1. 更新 progress.md 记录进度
2. 检查是否需要更新其他文档（对照触发条件表）
3. 更新 INTERRUPT.md 记录当前状态
```

## 关键机制

### 文档更新触发条件

| 场景 | 必须更新的文档 | 更新内容 |
|------|---------------|----------|
| 新增功能需求 | requirements.md | 需求描述、优先级、验收标准 |
| 确定技术方案 | architecture.md | 架构设计、模块关系、接口定义 |
| 开始新迭代 | iteration-plan.md | 迭代目标、任务拆解、时间线 |
| 修改需求 | changes/requirements.md | 变更原因、影响分析 |
| 修改架构 | changes/architecture.md | 变更原因、兼容性说明、迁移方案 |
| 每轮对话结束 | progress.md + INTERRUPT.md | 当前进度、阻塞问题、下一步 |

### 关键文档 Review 机制（阻塞式）

以下文档在**生成或更新后**必须交付用户 Review，**通过后才能继续**：

| 文档 | 触发时机 | Review 要点 |
|------|----------|-------------|
| requirements.md | 生成或更新后 | 需求完整性、优先级合理性、验收标准可衡量 |
| architecture.md | 生成或更新后 | 架构合理性、模块划分清晰、接口定义完整 |
| progress.md | **首次**完成 Iteration 拆分后 | 任务拆解合理、工时估算可行、依赖关系清晰 |

**Review 流程**：
```
1. AI 完成关键文档更新
   │
   ▼
2. AI 明确提示用户：
   "【关键文档待 Review】{文档名} 已更新
   
   本次更新摘要：
   1. xxx
   2. xxx
   
   Review 检查点：
   - [ ] 检查点 1: xxx
   - [ ] 检查点 2: xxx
   
   请回复 "Review 通过" 或提出修改意见。
   ⛔ 当前工作已暂停，等待 Review 确认"
   │
   ▼
3. 用户 Review
   ├─ 通过 → 继续下一步
   └─ 不通过 → 记录修改意见，返回修改
```

### 重要文档提醒机制（非阻塞）

以下文档在**每次更新后**提醒用户检查，**不阻塞进度**：

| 文档 | 触发时机 | 提醒内容 |
|------|----------|----------|
| progress.md | 每次更新后 | 本次更新摘要、下一步行动 |
| decisions.md | 每次决策后 | 决策内容、原因、影响 |

**提醒格式**：
```markdown
【重要文档已更新】{文档名}

本次更新摘要：
- 要点 1: xxx
- 要点 2: xxx

不阻塞进度，您可在方便时 Review。
```

## 文档健康检查（强制）

**每次对话开始时，必须执行：**

### 检查清单
检查以下必须文档是否存在：
- [ ] progress.md
- [ ] iteration-plan.md
- [ ] requirements.md
- [ ] architecture.md
- [ ] decisions.md
- [ ] INTERRUPT.md

### 缺失提醒
如有缺失，主动提醒用户：
```
【文档检查】检测到以下文档缺失：{列表}
建议创建以完善项目管理。

需要我帮您创建 {文档名} 吗？
```

### 文档过期检查
检查 progress.md 最后更新时间：
- 如果超过 24 小时未更新，提醒：
  "【文档检查】progress.md 已 X 天未更新，建议同步最新进度。"

---

## 关键文档 Review 提醒（强制）

**当检测到以下情况时，必须主动提示 Review：**

### 触发条件
- 用户说"写完了"、"完成了"、"done" + 文档类型
- 用户请求"看看有没有问题"、"检查一下"
- 文档内容超过 500 行或 10 个章节
- 迭代进入"验证需求结构"阶段

### Review 提示模板
```
【关键文档 Review】
检测到 {文档名} 可能需要 Review。

Review 检查清单：
- [ ] 结构是否清晰
- [ ] 内容是否完整
- [ ] 是否有遗漏的边界情况
- [ ] 是否与其他文档冲突

请选择：
1. 开始 Review（我会逐条检查）
2. 跳过（标记为已 Review）
3. 稍后提醒
```

### 阻塞式 Review
**必须完成 Review 才能继续：**
- output_product_doc.md（最终产出文档）
- architecture.md 重大变更
- 迭代完成前的最终检查

---

## 最佳实践

1. **文档先行**：任何功能开发前，先更新需求/设计文档
2. **变更留痕**：需求或架构变更必须记录原因和影响到 changes/ 目录
3. **测试同步**：功能开发与测试用例编写同步进行
4. **进度透明**: 每次会话更新 progress.md
5. **阻塞即记录**: 遇到问题立即记录阻塞原因到 progress.md
6. **决策留痕**: 关键决策写入 decisions.md
